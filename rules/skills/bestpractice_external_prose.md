# 外部文章微观文风与修辞规范

## 元数据

- **类型**: BestPractice
- **适用场景**: 面向没有共享上下文的陌生读者、公开发布渠道、客户或外部课程受众的中文长文/报告/博客写作。
- **创建日期**: 2026-07-06
- **最后更新**: 2026-07-16

## 核心设计哲学

外部文章的第一原则是**降低陌生读者的决策与认知阻力**。在句子 and 词汇层面，必须彻底去除“AI腔”、“翻译腔”以及“浮夸的虚假共识”。

---

## 1. 禁用修辞、比喻与抽象词

对系统、抽象概念和分析判断，写清其演化机制与差异，禁止使用以下口语化、文学化或无具体指代标签：

*   **口语化动作比喻（全文禁用）**：禁令适用于整篇文章。任何位置都禁止使用“长出来/长出了”、“生长”、“很硬”、“很锋利”、“收口”、“很稳”、“更稳”、“击穿”、“打穿”、“拆解”、“闭眼编”。
    *   *改法*：用机制描述替代。如“击穿/打穿”改为“否定了该预设”、“边界失效”；“拆解”改为“分析”、“分成几层看”、“逐项检查”；“闭眼编”改为“未进行实时检索直接生成”。
*   **结构性**：禁止使用“结构性原因”、“结构性矛盾”等笼统词。
    *   *改法*：写出具体的底层机制、约束来源或冲突位置。
*   **叙事弧线**：禁止用“叙事弧线”、“弧线”（English: narrative arc）来形容文章或事件走向。
    *   *改法*：写明按什么顺序展开、围绕什么推进。
*   **系统状态模糊词**：少用“崩”、“撑不住”、“撑不起”概括系统或论证状态。
    *   *改法*：改为“异常终止”、“请求失败”、“会话漂移”、“前提不成立”、“难以维持”。

---

## 2. 译文体与词意错配纠偏

避免将英文词汇的语义范围生搬硬套到中文里。以下为高频词义错配与修正：

*   **expensive / cheap** $\rightarrow$ **贵 / 便宜**：中文“贵/便宜”仅用于具体商品价格。抽象代价应用“代价高/低”、“成本高/低”。
    *   *示例*：“工程执行太贵” $\rightarrow$ “工程执行代价太高”；“变便宜以后” $\rightarrow$ “成本降低以后”。
*   **natural** $\rightarrow$ **天然**：中文“天然”指非人造。固有属性应用“天生”、“自带”、“内在”。
    *   *示例*：“天然重要” $\rightarrow$ “天生重要”；“天然反馈” $\rightarrow$ “自带反馈”。
*   **model / framework** $\rightarrow$ **模型 / 框架**：中文“模型”在 AI 时代已被大模型（LLM）狭义占用。对于 mental model、judgment model 等，改用“思路”、“思维框架”、“评估方法”。
    *   *示例*：“判断模型” $\rightarrow$ “判断思路”；“mental model” $\rightarrow$ “思维方式/思维框架”。
*   **Metaphor 硬译**：
    *   “站在某一层” (stand on) $\rightarrow$ “属于哪一层”。
    *   “服务的是” (serves) $\rightarrow$ “针对的是”。
    *   “让人不舒服/不安的事实” $\rightarrow$ “和主流叙事相反”、“违反直觉”或直接陈述事实。
    *   “blast radius” $\rightarrow$ “判断半径”、“影响范围”。

### 禁止无信息增量的双语括注

中文已经表达清楚时，禁止紧接括号重复英文翻译，例如“记忆召回（memory recall）”“执行状态（execution state）”“检查点（Checkpointer）”。这类写法不增加信息，只增加视觉噪音和翻译腔，也是模型容易跨文章复用的模板。

处理方式：

* 普通概念直接保留中文，不附英文。
* 产品名、API 名、协议名、代码标识符和行业通用缩写保留官方英文，例如 `Turso Sync`、`push()`、`RLS`。
* 英文术语本身比中文译名更常用时，可以直接使用英文，并在句子中自然解释一次；不要写成“中文（English）”对照表。
* 只有当文章讨论译名差异、术语歧义或需要帮助读者检索原始资料时，才允许双语并列，并明确说明保留英文的原因。

### 避免长条件从句直译

中文并不禁用“当”，但要审查从英文 `When X, Y becomes...` 直译来的固定骨架：**“当 + 一长串条件 + 时/时候，……就会/成为/意味着……”**。这类句子常把真正发生的动作藏进从句，再用“成为一个明显的问题”“带来新的挑战”作抽象结尾。

标题没有真正的时间转折时，直接删掉“当”。正文改成“变化 → 人的动作 → 实际后果”的顺序，不要只把“当”换成“随着”或“在……情况下”。

*   ❌ **Bad**: 当开发者还需要逐个切换面板去查看它们时，如何分配注意力就成了一个明显的问题。
*   ✅ **Good**: Agent 多了以后，人还得一个窗口一个窗口地翻，看谁跑完了、谁卡住了、谁在等确认。任务跑得更多，管理方式却还停在单 Agent 时代。

### 避免机械的“可 X、可 Y、可 Z”属性排比

英文常把一组动作压成并列形容词。逐项翻成“可等待、可拒绝、可恢复”虽然语法成立，却把人在什么时候做什么改写成抽象属性。

描述流程时，写清执行者和动作先后，不要为了保留英文原句的对称结构强行凑排比。`可验证`、`可追踪`等词单独描述技术属性时可以保留；连续出现两个以上“可 + 动词”时，必须检查能否改成人和动作。

*   ❌ **Bad**: 协议把人的中途参与变成可等待、可拒绝、可恢复的状态。
*   ✅ **Good**: Server 发出请求后，当前任务先停在原地。用户可以补充信息，也可以拒绝或取消；client 把结果送回去，server 再从原来的位置继续。

## 3. 禁用的文章开篇、预设句式与全文表达

避免用虚假的共识或居高临下的语气来强拉读者进入：

*   **「如果你 X，大概率 Y」开篇**：禁止用此句式做文章或段落开头。它用虚假共识制造进入感，没有信息增量，纯属注意力消耗。
    *   *改法*：直接陈述事实，或用具体事件开篇。
*   **「如果你 X，你大概知道 Y」预设读者知识**：这会在不知不觉中排除不了解的读者，或对已了解的读者造成冗余。
    *   *改法*：直接陈述 Y，主语放在客观对象上。
*   **「值得 X」推荐语（全文禁用）**：禁令适用于整篇文章，不限于开头。任何位置都禁止使用“值得认真对待/值得深思/值得关注/值得一提/值得记住”等“值得 X”表达。这属于居高临下的评判。
    *   *改法*：删掉标签，让事实和机制自己呈现（例如，用“容易被混淆的地方”代替“值得分开看的地方”）。

---

## 4. AI 写作模板化修辞降级

AI 极易重复使用以下高频修辞模板，需进行降级或替换：

*   **「不是 X，是 Y」对照框架**：单文出现 $\ge 3$ 次必须重写。
    *   *改法*：直接陈述 Y 是什么；或者改为轻否定承接（“没有 X，只是 Y”、“与其说 X，不如说 Y”、“表面看是 X，实际约束在 Y”）。
*   **抽象主体拟人化**：禁止将建筑、年份、街区等配以“写给”、“接住”、“画线”等动作。
    *   *改法*：还原为具体人/主体的具体动作。
*   **排比句式套用**：避免“一份条例定义材料，一种风格定义构图...”之类的机械排比，拆成正常句子。

## 5. 写作微观负例对 (Bad-to-Good)

在撰写外部文章时，主线程或写作 Agent 应参考以下负例对来校准句式和词汇：

### 负例对一：长定语拆开
*   ❌ **Bad**: 那些单看耗时在效率表上可能像是低效环节的搬运、整理和样板工作，在 AI 时代成本降为零。
*   ✅ **Good**: 这些事单独看不值钱。老板看时间表，可能会觉得它们是低效环节。但对人的大脑来说，它们有用。

### 负例对二：端着词与抽象主语降级
*   ❌ **Bad**: 该项技术的发布带来了更强烈的疲惫感，显著关联了用户的认知超载，背后的机制极其重要。
*   ✅ **Good**: 新技术出来后，人却更累了。这说明用户的大脑超载了。

### 负例对三：不是/而是转折降级
*   ❌ **Bad**: 你不是做了更多体力活，而是更长时间停留在高认知带宽状态里。
*   ✅ **Good**: 你没有做更多体力活，只是更长时间停留在高认知带宽状态里。

### 负例对四：开篇 Hook 降级
*   ❌ **Bad**: 如果你在 2026 年初关注 AI 工程，大概率听过 Multi-Agent 是大趋势的判断。
*   ✅ **Good**: 2026 年一开年，AI 工程圈就被两股相反的力量挤在中间：一边是 Swarm 在 benchmark 上拉出新高，另一边是实际框架的失败率居高不下。

## 6. 反教科书腔：让解释像人在说话

低认知负担不只是不难懂。很多句子事实准确、语法简洁、概念齐全，读起来仍像产品手册或教科书。典型症状是：先给对象下定义，再宣布“这一调整产生了某种影响”，随后用抽象名词概括讨论。读者每句话都看得懂，却感觉不到作者在带他一起发现问题。

亲切感也不等于加语气词、网络梗或第二人称。它来自一种更基础的写法：作者和读者面对同一个具体问题，作者把自己看到的异常、疑问和推导顺着讲出来。概念在需要时顺手解释，而非排队接受定义。

### 6.1 禁止把“概念引入”写成词典释义

下列句式不是绝对禁用，但在一段内连续出现时，通常说明文章退回了说明书语气：

- “X 是一款/一种……，允许用户……”
- “X 指的是……，其主要作用是……”
- “这一调整改变了/影响了……”
- “这引发了业界对于 X 的讨论。”
- “在这一过程中/在该体系下/从这一角度来看……”
- “这为后续的 X 提供了基础。”

修改时不要只换同义词。先问：读者此刻真正想知道什么？谁做了什么？哪里和以前不一样？出了问题以后，人具体少了哪一种判断能力？把答案写出来，概念自然会被引入。

### 6.2 用动作和疑问承载概念

*   ❌ **Bad**: Codex 是一款开源的本地编程代理客户端，允许开发者在本地环境调用 OpenAI 的大模型来执行任务。
*   ✅ **Good**: Codex 会在你的电脑上读写文件、运行终端命令，也会把执行记录保存在本地。出了问题，开发者通常会顺着这些记录往回查。

*   ❌ **Bad**: 这一调整直接影响了本地审计。
*   ✅ **Good**: 麻烦出在回头查问题的时候。你知道主 agent 开了一个 sub-agent，也看得到 sub-agent 后来做了什么，偏偏中间那句“你去做什么”只剩下一串密文。

*   ❌ **Bad**: 这引发了技术社区对代理透明度的讨论。
*   ✅ **Good**: 所以 issue #28058 里大家追问的其实很具体：这个 sub-agent 到底收到了什么任务？它后来做错了，是任务派错了，还是自己执行错了？

这些 Good 示例展示的是转换方法，不是新的固定模板。不要在每篇文章里机械复用“麻烦出在”“大家追问的其实很具体”或任何一组现成措辞。

### 6.3 句子要有呼吸，不要全部切成说明书短句

短句能减轻局部负担，但连续的“主语 + 是/会/能 + 定义”会形成播报感。自然中文通常长短句混合：短句负责落判断，稍长的句子负责把原因、例子和转折连起来。不要为了满足字数上限，把本来连贯的一段切成五句同样长度的陈述。

判断标准不是统计平均句长，而是出声读时像不像一个熟悉这个问题的人，正在给另一个聪明人解释。如果每句话都可以单独放进产品文档，却没有一句像真实对话中的解释，就需要整段重写，而不是继续拆句。

### 6.4 抽象名词必须落到读者能观察的后果

“透明度下降”“审计受阻”“信息边界变化”都可以作为段落结论，但不能代替解释。第一次出现时必须落到具体后果，例如：

- 看得到 agent 被创建，看不到它收到的任务；
- 看得到命令执行，看不出命令来自错误委派还是错误理解；
- 能重放工具记录，不能重建完整的因果链。

先让读者看到这些事实，再给它们命名。不要先抛出“可审计性下降”，再用三段话解释这六个字是什么意思。

### 6.5 亲切中文不是口语化，而是缩短叙述距离

文章有时已经没有长句和生僻词，仍然显得端着。原因往往不是词太难，而是叙述者站得太远：它替系统创造了一套角色名称，用机构视角描述动作，再用抽象名词总结后果。真实的人解释同一件事时，通常会直接说“你会看到什么”“哪里少了一步”“出了问题该怎么查”。

重点检查四类翻译腔：

- **多余的角色名词**：`客户端及其操作者`、`任务接收方`、`执行主体`。读者身份明确时直接写“你”“开发者”，或者删掉主语。
- **名词化动作**：`对该状态进行加密`、`完成任务的分发`、`实现明文的恢复`。改回普通动词：“加密这段内容”“把任务发出去”“恢复明文”。
- **系统视角的结果句**：`使本地日志无法显现其具体内容`。先说发生了什么，再说读者看到什么：“这部分返回本地时已经是密文，所以你从日志里看不到具体内容。”
- **抽象后果代替真实损失**：`失去关键因果输入`、`缺少直接观测点`、`无法进行前置校验`。直接写缺了哪条信息、因此分不清什么。

下面三组改写来自同一篇实际文章。它们展示的是视角转换，不是固定句式：

*   ❌ **Bad**: 运行在本地的 Codex 客户端及其操作者，无法再从本地记录中读取这段委派任务的文本。
*   ✅ **Good**: 少掉的是你回头查看这句话的能力：主 agent 当时究竟让它做了什么，本地记录不再告诉你。

*   ❌ **Bad**: Responses API 会在返回数据时对该状态进行加密，使得本地日志无法显现其具体内容。
*   ✅ **Good**: 这部分内容返回本地时已经是密文。你能看到最后的回答，却不能从日志里读出它中间具体想了什么。

*   ❌ **Bad**: 当指令文本在本地变成密文后，开发者失去了这一因果输入的直接观测点。
*   ✅ **Good**: 你仍然看得到 sub-agent 何时被创建，也看得到它后来运行了哪些命令。偏偏中间那句“你去做什么”只剩下一串密文。

最终判断不是“是否用了第二人称”，而是一个熟悉问题的人会不会这样向聪明读者解释。不要为了显得亲切而加入俚语、网络梗、夸张比喻或过多语气词；这些只会把正式腔换成表演式口语。

### 6.6 不要把编辑对话写进正文

编辑或用户指出一个困惑时，这个问题通常说明正文缺少前置信息，不代表文章应该原样回答它。最常见的坏法，是把编辑的误解直接写成标题或否定句。新读者没看过那段对话，会突然看到作者反驳一个从未出现的观点。

*   ❌ **Bad**: 选择 1.7B 不是在两个训练结果中挑赢家。
*   ✅ **Good**: 每份训练作业在 GPU 开跑前都要指定一个起始 checkpoint，可选项是 0.6B 和 1.7B。后期选择 1.7B 的作业比例上升，说明训练预算的分配发生了变化。

*   ❌ **Bad**: prime-rl 不是另一个被选择的模型。
*   ✅ **Good**: 起始模型确定后，prime-rl 负责按作业里的配置执行内层强化学习训练。

修改后做“隐藏对话测试”：删掉所有编辑记录，只读正文。读者是否已经从前文知道这句话在回答什么？如果不知道，应补足动作顺序、角色和影响，而不是保留针对误解的反驳句。

### 6.7 目标不是口语化，而是自然介绍

外部技术文章最常见的文风误判，是把“准确、易懂”当成“亲切”。一篇文章可以没有长句、没有生词、每个概念都解释清楚，读起来仍像教材。原因通常在段落架构：作者先宣布本节主题，再定义对象，随后解释机制，最后用抽象判断收束。读者接收到的是一堂经过简化的课，不是一个人正在分享自己刚看懂的变化。

合格的自然介绍通常按另一种方式推进：

1. 先说本篇特有的反差、疑问或观察，让读者知道作者为什么此刻想讲这件事。
2. 沿一个真实过程往下走。概念在动作发生时出现，而不是按术语表顺序出现。
3. 作者的判断可以轻量进入，例如“我觉得这次变化最有意思的地方……”。第一人称只说明关注点，不渲染情绪，也不替代证据。
4. 每一节回答读者读到这里自然会追问的下一件事。段落落到谁要等、什么要搬、哪项能力会丢、成本算在哪里，而不是每节都宣布一次行业意义。

下面三种声音要分开：

*   **教材声**：`原因藏在一次生成请求的两个阶段里。调度器因此要在两类目标之间取舍。` 事实没错，但作者在按提纲授课。
*   **表演式口语**：`vLLM 就像一辆大巴车。几个阶段会互相掐架。` 句子更活泼，却用夸张比喻和俚语替代了机制，还可能诱发事实过度简化。
*   **自然介绍**：`vLLM 想让整套机器多干活，TileRT 想让眼前这个请求更快。为什么一个引擎不能把两边都做好？问题要从模型生成一段回答的过程说起。` 作者在场，问题具体，后文也有自然的因果入口。

这些例句展示叙述距离，不是可复制模板。新文章必须从自己的证据中找到入口，不能机械复用“原本不像会走到一起”“最有意思的地方”“麻烦就在这里”等句式。

#### 双向校准法

每次 prose pass 都要同时准备一个正向样本和两个负向极端：

* 正向样本选用户明确认可、且题材解释难度相近的文章。可选一篇你自己确认过的技术文章作为 voice reference。
* 教材负样本从本次结构稿或 IC-2 初稿中摘 2-3 段，标出“定义 → 机制 → 总结”的重复节奏。
* 过度口语化负样本摘 2-3 段，标出装饰性比喻、俚语、拟人和绝对化判断。

校准时只学习正向样本的叙述距离：作者如何进入、问题如何自然出现、概念如何跟着动作落下。不得模仿其固定开头、第一人称位置、比喻或段落数量。

#### 一分钟出声检查

全文完成后，连续读开头、每个 H2 的第一段和结尾。逐段判断：

* 一个真正理解此事的人，会不会这样向聪明朋友开口？
* 删掉标题后，这一段是否仍像在宣布“本节将介绍什么”？如果是，重写入口。
* 删掉比喻或俚语后，机制是否还说得清？如果不清楚，说明亲切感建立在表演上。
* 作者是否只在有判断时出现？如果“我觉得”后面只是事实摘要，删掉第一人称。

这项检查优先级高于平均句长、连接词数量和“口语词”命中数。后几项都合格，不能抵消教材声或表演声的一票否决。

### 6.8 看得懂不等于读得舒服

低认知负担有两个不同层面。第一，读者是否具备理解每个概念所需的上下文。第二，读者在任意时刻需要同时记住多少个新概念和关系。前者通过，后者仍可能失败：每个术语都有解释，整段也没有长句，读者却要停下来整理作者刚列出的四条轴、三层对象和两组例外。

最常见的原因是把作者的分析过程直接变成正文。作者为了想清楚问题，可能需要稳定身份、状态归属、扩展单位、恢复单位四条检查轴；读者未必需要先学会这套框架。除非文章的交付本身就是教读者使用该检查表，否则应先让一个具体对象沿真实过程走完，读者看到差异后再给它命名。

*   ❌ **Bad**: 看懂一个平台，需要同时检查稳定身份、状态归属、扩展单位和恢复单位。
*   ✅ **Good**: 同一张客服工单，在一种平台上由新启动的程序从数据库读回来；在另一种平台上，下一条消息凭 Agent ID 找回原来的局部状态；在第三种平台上，审批回来后，任务从上次停下的步骤继续。

Good 版本不是因为词更口语，而是因为四条抽象关系被一个熟悉对象按时间顺序承载。读者只需跟着同一张工单往前走，不必同时操作一张分析矩阵。

降低这类负担时遵循四条原则：

* **同一个过程优先**：横向比较多个方案时，尽量让同一个请求、人物、订单或文件贯穿各节。每节更换场景会迫使读者重新建立上下文。
* **动作先于命名**：先写“批准回来后从下一步继续”，再引入 Workflow；先写“下一条消息回到同一个客户助手”，再引入 Agent ID。
* **标题可以冷读**：H2 单独出现时也应表达动作、变化或读者问题。只有掌握文章内部 taxonomy 才能理解的标题，会在读者进入正文前先收取一次概念成本。
* **一段推进一个主关系**：拆短句不能解决概念堆叠。若一段同时引入平台、组件、状态模型和故障边界，应重新安排信息顺序，让每段从读者已经认识的对象开始。

最终做复述测试：读完一节第一段，不使用刚出现的专业词，能否说清谁做了什么、哪里变了、为什么与前一方案不同？如果只能复述分类标签，说明文章仍把作者的框架交给了读者。
